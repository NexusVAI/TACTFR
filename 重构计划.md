# TACTFR 重构计划（给未来的我/你）

> 目标：把现在“功能互相污染、状态紊乱”的链路拆成可扩展的模块，支持：  
> - 拘捕风格：**被拷线** / **举手线**（两套逻辑不互相污染）  
> - 案件模式：**步行嫌疑人** / **驾车逃逸嫌疑人**（两套流程不互相污染）  
> - 人数：**单人案件** / **双人案件**（两名嫌疑人可以不同拘捕风格；G/E 统一指令但按 handle 分开执行）  

本文档写于：2026-02-21（周六）。

## 1. 玩家交互规则（必须保持）

### 1.1 键位前置链（硬规则）

- **L**：锁定目标（LockTarget）  
  - H 的前置：必须先 L 锁定（当前代码已强制）
- **H**：打开拘捕动作菜单并执行拘捕（Arrest）  
  - G 的前置：必须已经完成 H 拘捕（嫌疑人被判定为 Arrested/Restrained）
- **G**：让已拘捕嫌疑人开始跟随/押送（Follow/Escort）  
  - E 的前置：必须先 G（FollowRequested=true）
- **E**：只负责“要求嫌疑人上下车”（VehicleInteract）  
  - E **不**承担“解拷/进入押送/其它杂项”，避免按键抢占与逻辑分叉

### 1.2 两条拘捕线的核心差异

#### 被拷线（CuffAndLead）
- 嫌疑人永远双手在背后被拷
- E 上下车时：系统负责 **自动开门/关门**（最佳体验：腰一扭屁股坐进去即可）

#### 举手线（HandsOnHeadFollow）
- 嫌疑人是“抱头/举手”状态跟随
- E 上下车时：嫌疑人 **自己开关门**（不走自动门逻辑；最多 best-effort）

### 1.3 双嫌疑人交互

- 两名嫌疑人允许 **不同拘捕风格**
- **G 统一召唤**：对已拘捕的嫌疑人下达跟随
- **E 一个一个上车**：对“当前锁定的那一个嫌疑人”上下车
- **全局门禁规则**（当前你描述的现状/期望）：
  - 双人案件里只要有人未拘捕，则另一个即便已拘捕也可能不动（由你最终决定是否保留为设计规则；建议未来做成“明确的 case-level gate”，而不是被 currentSuspect 副作用卡住）

## 2. 当前工程的关键现状（为什么会乱）

### 2.1 “单例 currentSuspect”是双人污染源

`SuspectController` 目前以 `_currentSuspect` 为中心（单例），并且 `CurrentArrestStyle` 也是“当前嫌疑人/全局”的概念。  
双嫌疑人要做到“每人不同风格 + 独立状态机”，必须把“风格/状态/忙碌/跟随标记”等运行态改为 **per-handle**。

### 2.2 `SuspectStateHub` 当前是单实例

`EFCore` 里：
- `_suspectStateHub = new SuspectStateHub();` 只有一份  
这会导致“一个人 EnteringVehicle/InVehicle”影响另一个人（如果后续真的把双人行为接进来，必炸）。

### 2.3 `SuspectVehicleEscortExecutor` 过度负责

它现在同时处理：
- E 意图（Enter/Exit）
- G 跟随逻辑
- 被拷线/举手线差异（夹在 if/else）
- PullOver 旁路
- 车门开关、座位选择、状态迁移、去抖

结果是任何新功能都容易引发“状态机/输入抢占/互相污染”。

## 3. 我们的重构目标架构（最终形态）

### 3.1 核心对象：每个嫌疑人一份 `SuspectRuntimeContext`

每名嫌疑人独立持有：
- `Handle`
- `ArrestStyle`（被拷线/举手线）
- `StateHub`（每人独立状态机）
- `FollowRequested`（E 的前置）
- `Busy` / `LastCommandAtMs`（防抖与中间态门禁）

### 3.2 案件对象：CaseManager 只负责“嫌疑人列表 + 只读状态”

CaseManager 已有：
- `SuspectHandles`（列表）
- `CurrentSuspectHandle`（primary 概念）
- `ISuspectContext`（对外只读视图）

未来要补：
- `PerHandle` 的运行态不应该继续塞 CaseManager 字段里，而是交给 SuspectRuntimeContext/Registry。

### 3.3 线路策略：`IArrestLine`（被拷线/举手线分治）

把两条线拆成两个策略类，各自实现：
- `OnFollowRequested(ctx, player)`（G）
- `OnVehicleInteract(ctx, player)`（E）
- `Tick(ctx, player)`
- `AutoDoorPolicy`（被拷线自动门，举手线不自动门）

这样可保证“完全走不同逻辑，不共用就不污染”。

## 5. 下一步重构路线（Step2~Step6）

### Step2：把“风格来源”切到 per-handle（不再依赖全局 CurrentArrestStyle）
- 新增一个统一入口：`GetArrestStyleFor(handle)`（优先 registry，fallback 旧逻辑）
- 把 `SuspectVehicleEscortExecutor.GetStyle()` 等读取点改用 per-handle（逐步替换）

### Step3：把“跟随前置(G)”与“E前置(G)”变成 per-handle
- 当前 `_isSuspectFollowing/_followingSuspectHandle` 是全局字段，会污染双人
- 改为：每个嫌疑人的 ctx 有 `FollowRequested`

### Step4：把状态机 `SuspectStateHub` 变成 per-handle
- 当前只有 `_suspectStateHub` 一份
- 双嫌疑人必须拆成：`ctx.StateHub`
- ⚠️ 风险提示：StateHub 直接控制行为；同时运行“两个中心”会产生冲突，所以必须用 **Router + 渐进迁移**，禁止并行写状态。

#### Step4a：为 StateHub 引入“句柄归属”构造
- 新增：`SuspectStateHub(int suspectHandle)`（或 `SuspectStateHub(SuspectKey key)`）
- 约定：每个 hub 只服务一个 handle（用于日志、断言与 Router）

#### Step4b：`SuspectRuntimeContext` 创建时拿到“自己的 hub”
- `new SuspectRuntimeContext(handle)` 内部创建 `new SuspectStateHub(handle)`
- 旧的 `_suspectStateHub` 暂时保留，但只作为 Router 的 fallback

#### Step4c：新增 `StateHubRouter`（决定“这个 handle 用哪一个 hub”）
- Router 提供：
  - `SuspectStateHub GetHubFor(int handle)`
  - `SuspectStateHub GetLegacyHub()`（只读/只用于 fallback）
- 行为：
  - 如果 registry/ctx 中存在 per-handle hub，则返回它
  - 否则回退到旧的单例 hub
- 说明：Router 的模式要和 Step2 的 `StyleResolver` 一致（优先新 -> fallback 旧）

#### Step4d：按状态逐个迁移（一次只迁一个状态的“写入权”）
- 原则：任何时刻，**同一个状态只允许一个中心写入**（要么旧 hub 写，要么 per-handle hub 写）
- 推荐迁移顺序（按风险从低到高）：
  1. `Escorting/Following`（跟随/押送）
  2. `EnteringVehicle`（上车中）
  3. `InVehicle`（车内）
  4. `ExitingVehicle`（下车中）
  5. `Restrained` / `Resisting` 等更敏感状态（如存在）
- 每迁移一个状态：把对应事件映射与 executor 的查询都切到 Router

#### Step4e：每次迁移后立刻回归测试
- 按第 6 节测试清单跑一遍（尤其：单人步行/单人驾车/双人混合风格）
- 只要出现“状态跳变异常/重复执行任务”，立刻回滚该状态迁移点，先补 Router 日志/断言


### Step5：拆 `SuspectVehicleEscortExecutor` 为“输入意图”与“线路策略”两层
- 输入层只接收事件（G/E），解析 handle，然后交给对应 `IArrestLine` 执行
- 线路策略决定：自动门、座位、Enter/Leave 的具体 task

### Step6：恢复双嫌疑人案件入口（FeatureGates.EnableDualSuspectCase）
- 先只恢复生成与追踪，不开复杂行为
- 然后逐步接入：G 统一召唤、E 单个上下车、风格独立、交付判定等

## 10. 必要的“安全护栏”（强烈建议补充并长期维护）

> 你担心“我以后忘了上下文/写重复/引入冲突”，这里给文档加一组硬约束：每次改动都对照这几条。

### 10.1 全局唯一原则（Single Writer / Single Source of Truth）

- **每个 handle 的风格来源只有一个**：`SuspectStyleRegistry`（或未来的 `SuspectRuntimeContext.ArrestStyle`）  
  禁止：同时从 `ArrestStyleState.SelectedStyle`、`SuspectController.CurrentArrestStyle`、某个 Executor 私有字段多头读取。
- **每个 handle 的状态机只有一个“写入者”**：迁移期由 `StateHubRouter` 决定写入目标。  
  禁止：同一状态同时由 legacy hub 与 per-handle hub 写入。

### 10.2 输入事件的责任边界

- `InputManager` 只负责“按键 -> 事件”，不直接下 Task，不直接改状态机。
- 行为执行器（Executor/LineStrategy）只响应事件与状态变化，不直接读键盘。

### 10.3 迁移期的日志与断言（最重要）

每迁移一个状态（Step4d），必须加：
- Router 选择日志（handle -> hub）
- 状态写入日志（old->new + handle + writer）
- 如果检测到“同 handle 同帧多次写入同一状态”，直接 `ModLog.Warn`（必要时强制忽略重复写入）。

### 10.4 防重复改动的工作流（避免你说的“经常写重复”）

- 每次开始改一个文件前：先 `find` 或全文搜索关键字段（例如 `_suspectCtxRegistry`）确认是否已经存在。
- 每次提交一个 Step：只允许影响 **一个功能点/一个文件块**（小步快跑），并在文档的“StepX”里记录“已完成/未完成”。

ArrestLine（拘捕线策略）：被拷线/举手线做成两个策略类（各管各的 Follow/Escort/Vehicle 逻辑），共享的只有接口，不共享内部状态。
Scenario（步行/驾车逃逸）：只负责“生成嫌疑人/触发阶段事件（逼停、下车、举手）”，不要写 E/G 的具体行为。

0) 先说结论：现在最大的耦合点在哪
SuspectController 只有一个 _currentSuspect + 全局 CurrentArrestStyle：这天然不支持“双嫌疑人各自风格”，也会导致你说的“共用污染”。
SuspectStateHub 看起来也是单实例：双嫌疑人需要“每人一份状态机”，否则一个人进入 EnteringVehicle 会把另一个人也拖进去。
SuspectVehicleEscortExecutor 现在既当“输入意图处理器”，又当“行为执行器”，还塞了 pull-over 旁路、门逻辑、follow gating —— 所以一加新类型就炸。
你要的解耦，本质就是：把“案件/嫌疑人数据”变成多实例，把“线路逻辑”做成策略，把“输入”只发意图。

1) 我建议的最终结构（按你术语命名）
A. 每个嫌疑人一个上下文：SuspectContext（核心）
每个嫌疑人（handle）独立拥有：

Handle
ArrestLine（被拷线 / 举手线）—— 每人独立
SuspectStateHub（None/Restrained/Escorting/EnteringVehicle/InVehicle/ExitingVehicle…）—— 每人独立
Busy / Following / LastCommandAtMs 等运行态
（可选）VehicleSeatReservation：谁占哪个座，防止抢座位
工程里其实已经有雏形：ISuspectContext + CaseSuspect + SuspectHandles（CaseManager）——这套可以作为“案件级只读视图”，但还缺“按 handle 的 per-suspect 上下文”。

B. 两条拘捕线做成策略：IArrestLine
两条线完全不共用内部状态，只共用接口：

OnFollowRequested(ctx, player)：G 的行为（统一召唤过来）
OnVehicleInteract(ctx, player)：E 的行为（只上下车）
Tick(ctx, player)：自动上车/自动下车/关门等
ShouldAutoDoor：被拷线 true，举手线 false
这样“被拷线自动开关门 / 举手线自己开关门”就自然落在策略里，不会在一个 executor 里用 if (style==...) 到处污染。

C. 案件类型（步行/驾车逃逸）只负责“阶段事件”，不碰 E/G
步行：生成嫌疑人、允许 H 菜单选线，然后“拘捕成功 -> 嫌疑人进入 Restrained”
驾车逃逸：PullOver 流程完成后，发一个“嫌疑人已合规/可押送”的事件，把嫌疑人放进 Restrained，但不要在这里写上车逻辑（上车永远由 E 触发）。
你现在 LockTargetSystem.AutoLockAndArrest/AutoLockCompliant 已经有“逼停后直接放行 G/E”的思路，这块以后可以保留，但“放行”应该变成：把对应嫌疑人 ctx 的状态推进到 Restrained，而不是到处打补丁。

2) 交互规则按你的设定落地（以后不容易乱）
G（统一召唤/跟随）
对 本案所有已拘捕（Restrained/ Escorted…）嫌疑人 下达跟随（两人一起过来）
每个嫌疑人 ctx 自己维护 IsFollowing=true（不要再用全局 _isSuspectFollowing）
E（只上下车，且前置 G）
对“当前锁定嫌疑人”执行上下车（一个一个上）
前置条件：该嫌疑人 ctx 必须 IsFollowing==true（也就是你说的 E 前置 G）
被拷线：E 会自动开门、塞入、上车后自动关门
举手线：E 只下达 EnterVehicle/LeaveVehicle，不做自动门（或只做 best-effort 不强制）
3) 双嫌疑人“不同风格”怎么存（关键）
不要再用全局 ArrestStyleState.SelectedStyle 当运行时权威；它只能用在“菜单那一刻的选择”。
真正权威应该是：每个嫌疑人 ctx 上的 ArrestLineType（或 ArrestActionStyle）。
